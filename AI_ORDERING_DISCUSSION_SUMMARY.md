## 一、项目概述

基于 Spring Boot WebFlux 和 DeepSeek 大模型构建的智能点餐系统，支持自然语言点餐和智能菜品推荐。

### 技术栈
| 组件 | 技术 | 版本 |
|------|------|------|
| 框架 | Spring Boot + WebFlux | 3.2.5 |
| 数据库 | H2 + R2DBC | 响应式 |
| 大模型 | DeepSeek Chat API | - |
| HTTP客户端 | OkHttp | 4.12.0 |
| 语言 | Java | 21 |

---

## 二、核心功能实现

### 1. AI自然语言点餐
- 解析用户自然语言输入
- 提取菜品名称和数量
- 匹配数据库中的菜品

```bash
curl -X POST http://localhost:8080/api/ai/order/parse \
  -H "Content-Type: application/json" \
  -d '{"input":"我要一份宫保鸡丁和两份鱼香肉丝"}'
```

### 2. 智能菜品推荐
- 根据用户偏好推荐菜品
- 调用DeepSeek大模型生成推荐理由

### 3. 响应式架构
- 使用Mono/Flux进行异步数据流处理
- R2DBC实现非阻塞数据库访问

## 三、关键技术讨论

### Q1: 响应式编程能加快响应速度吗？

**A:** 不能。响应式编程的核心价值在于：

| 特性 | 说明 |
|------|------|
| **高并发处理** | 少量EventLoop线程处理大量并发请求 |
| **资源利用率** | 线程在等待IO时可处理其他请求 |
| **背压控制** | 消费者控制数据流速，防止内存溢出 |
| **异步组合** | 优雅地组合多个异步操作 |

**关键结论**：响应式不会让单个请求变快，但能让系统在高并发下保持稳定和高效。

### Q2: 响应式与AI结合的原因是什么？

**A:** 主要基于以下几点：

1. **AI API调用是IO密集型操作**：网络请求会阻塞线程，响应式避免资源浪费
2. **AI响应可能是流式的**：现代大模型支持流式输出，Flux原生支持
3. **复杂AI工作流需要组合**：Mono/Flux提供强大的异步组合能力
4. **弹性伸缩**：少量线程处理大量并发AI请求

### Q3: 大模型支持流式输出是什么意思？

**A:** 流式输出是指大模型**实时、分段**地推送响应：

```
传统方式: 等待完整响应(2-5秒) → 返回全部内容
流式方式: 立即返回首字符(<100ms) → 持续返回片段 → 完成
```

**优势**：
- 更低的感知延迟
- 更好的交互体验（打字机效果）
- 更高的资源效率
- 支持中途中断

### Q4: 流式输出会导致逻辑错误吗？

**A:** 可能，但概率很低，且有机制保障：

| 保障机制 | 说明 |
|----------|------|
| **自回归生成** | 每个token基于前面所有内容预测 |
| **温度参数** | 控制随机性（低温度更确定，高温度更多样） |
| **模型训练** | 学习逻辑一致性和语境连贯性 |
| **RLHF优化** | 通过人类反馈提升输出质量 |

**本项目配置**：`temperature=0.7`，平衡创造性和准确性。

## 四、配置说明

### DeepSeek API配置
```yaml
ai:
  deepseek:
    api-key: ${AI_DEEPSEEK_API_KEY:}
    base-url: https://api.deepseek.com/v1
    model: deepseek-chat
```

### 运行方式
```bash
# 开发模式
mvn spring-boot:run

# 打包运行
mvn clean package
java -jar target/ai-ordering-agent-1.0.0-SNAPSHOT.jar
```

### 服务地址
- API服务：http://localhost:8080
- H2控制台：http://localhost:8080/h2-console
